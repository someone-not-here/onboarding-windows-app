### Why are PRs important in a team workflow?
Pull requests allow changes to be reviewed before they are pushed into the main branch. This ensures that any potential problems may be caught beforehand by reviewers.

### What makes a well-structured PR?
A well-strucuted PR should:
- Have a clear and descriptive title.
- Have a summary of changes and reasonings.
- Small and focused on a single task.
- References any related issues.

### What did you learn from reviewing an open-source PR?
I reviewed this pull request from the markitdown repo: https://github.com/microsoft/markitdown/pull/1551

I've observed that:
- A good pull request has a clear and descriptive title, has a summary of changes and context/reasoning for the changes, and references issues that the pull request tackles.
- Developers can offer alternative changes from what the pull request proposes.
