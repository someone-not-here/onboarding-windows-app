### What makes a good commit message?
A good commit message is clear, concise, and informative; it should explain every change and the reasoning behind them without being long-winded. They should be short, not vague or generic, and straight to the point.

An example of a good commit message would something like: 
- Title: "Fixed readability of annotation text in Habits section"
- Description: "Annotation text was long and in a single paragraph. Broke the text up so it is more readable."

### How does a clear commit message help in team collaboration?
Clear commit messages help others within the team quickly understand what was changed in the commit without having to ask whoever made the commit for an deeper explanation. This helps save time.

### How can poor commit messages cause issues later?
Vague commit messages such as "Updated file" or "Fixed bug" do not convey what the actual change in the commit is. This slows things down as others within the team have to likely ask whoever made the commit to explain what was changed and why.

Overly long and cluttered messages also slows things down as they can obscure relevant information to reviewers.
